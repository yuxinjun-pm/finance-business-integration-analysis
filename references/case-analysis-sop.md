# Case Analysis SOP

Use this reference when the user asks for a finance-business case analysis, profitability diagnosis, management accounting answer, or assignment-ready structured response.

## Core rule

Do not skip from case facts to recommendations.

Always move through:
metric boundary -> calculation -> total result -> segmentation -> driver diagnosis -> quantified actions -> implementation implications

## Step 1: Define the analysis metric

Before calculating anything, state:
- what profit metric you are analyzing
- what costs are included
- what costs are excluded
- whether the result is contribution profit or accounting profit

If the case provides activity cost rates or explicit allocation rules, use them as given.
Do not invent new allocations unless the task explicitly requires assumptions.

## Step 2: Compute at the object level

If there is usable data, calculate by the relevant object:
- customer
- product
- order
- channel
- business line

At minimum, try to get:
- revenue
- direct cost
- service/activity cost
- profit amount
- profit margin

## Step 3: State the total result first

Before discussing causes, show the total picture:
- total revenue
- total margin/profit pool
- profitable count vs unprofitable count
- how much the loss-making objects erode the total pool

Purpose:
- determine whether the issue is marginal or structural
- avoid overreacting to anecdotal examples

## Step 4: Compare groups

Do not just label objects profitable/unprofitable. Compare the groups.

Typical dimensions:
- average revenue
- gross margin
- service cost rate
- profit margin
- low-margin mix
- standardized behavior intensity

When different objects have very different scale, normalize behavior by a common denominator such as:
- per order
- per shipment
- per $100,000 revenue

## Step 5: Identify the true drivers

Prefer specific drivers over broad labels.

Possible drivers:
- low-margin product mix
- design intensity
- expedite intensity
- support burden
- order fragmentation
- shipment fragmentation
- capacity bottleneck
- pricing weakness
- strategic cost drivers
- execution cost drivers

Rank drivers when possible:
- biggest difference
- highest cost impact
- best action leverage

## Step 6: Translate diagnosis into actions

Every recommendation should correspond to a driver.

Examples:
- low-margin mix -> reprice, shift mix, set minimum margin thresholds
- high design intensity -> free design limits, overage fees, modular templates
- high expedite intensity -> expedite fees, gatekeeping, demand forecasting
- fragmented orders -> minimum order size, order bundling, fixed ship dates

## Step 7: Quantify the actions

Whenever data allows, express actions as:
- break-even price increase
- target reduction in design events
- target reduction in expedites
- minimum service fee
- segment-specific profitability threshold

Good formulas often look like:
- target profit = current profit + price increase impact + avoidable cost reduction
- break-even uplift = current loss / customer revenue
- required event reduction = current loss / unit activity cost

## Step 8: Separate improve vs exit

Never collapse these decisions.

For improvement decisions:
- use cost rates and action levers freely

For exit decisions:
- check avoidable cost only
- discuss fixed cost stickiness
- discuss idle capacity
- discuss contract and strategic constraints

Negative contribution does not automatically imply immediate exit.

## Step 9: Segment for management

For stronger answers, convert the objects into management groups such as:
- high value high efficiency
- high value high input
- low value improvable
- low value high consumption

Each segment should map to:
- pricing logic
- service logic
- operating logic
- account-management logic

## Step 10: End with implementation sequence

Do not end with abstract advice only.

Close with a simple execution order, for example:
1. monthly calculation
2. automated segmentation
3. segment-based action
4. follow-up tracking
5. quarterly recalibration

## Standard answer test

A strong answer should let the user clearly see:
- what was measured
- how it was measured
- what the result was
- why it happened
- what should be done next
